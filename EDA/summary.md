# Ko-MIRACL EDA Summary

데이터셋: [`taeminlee/Ko-miracl`](https://huggingface.co/datasets/taeminlee/Ko-miracl)
EDA 생성 스크립트: `EDA/eda_ko_miracl.py`
축소 코퍼스 생성 스크립트: `extraction/build_reduced_corpus.py`

## 1. 데이터셋 구성

| config | split | 행 수 | 구조 |
|---|---|---|---|
| corpus | corpus | 1,486,752 | `_id`(문서#청크번호), `title`, `text` |
| default | train | 12,767 | `query-id`, `corpus-id`, `score` |
| default | dev | 3,057 | 위와 동일 |
| queries | queries | 2,761 | `_id`, `text`(질문) |

## 2. corpus 구조 통계

- 전체 청크(문단) 수: **1,486,752개**
- 고유 원문서 수(`_id`의 `#` 앞부분 기준): **437,373개**
- 문서당 평균 청크 수: **3.40개**
- 청크 텍스트 길이(샘플 약 5만 개 기준): 아래 히스토그램 참고 — 대체로 짧은 문단 단위로 쪼개져 있음

## 3. queries 통계

- 질문 개수: **2,761개**
- 질문 길이: 평균 **19.7자** (표준편차 10.9) — 대부분 10–30자의 짧은 한 문장 질문

## 4. qrels(관련도 판정) 통계

- train **12,767행** + dev **3,057행** = 총 **15,824행**
- score=1(관련) **2,520행** / score=0(비관련) **13,304행** → 관련:비관련 ≈ 1:5.3
- 판정에 등장한 고유 corpus-id: **12,601개** (그중 관련도>0인 고유 문서는 **2,105개**)
- 쿼리당 관련 문서 수: 평균 **2.33개**, 최대 **12개**

## 5. 우리가 추출한 축소 코퍼스 통계 (BEIR 스타일, 20만 청크)

qrels에 등장한 corpus-id만 모으면 검색 공간이 148만 개 중 1%대(qrels 판정 문서만 약 1.5만 개)로 줄어들어 "정답이 후보 풀에 반드시 있다"가 보장되는 인공적으로 쉬운 평가가 된다는 문제가 있어, 아래 방식으로 재구성했다:

- **qrels(train+dev)에 판정된 문서(score 상관없이 전부) 7,692개는 원문서 단위로 전부 포함** — 판정된 특정 문단 하나만이 아니라 그 문서의 형제 청크(문단)를 통째로 포함시킴(청크 124,837개)
- **나머지 예산은 판정 안 된 문서 중 무작위(seed 고정, 재현 가능)로 채움** — 23,822개 문서, 청크 75,163개
- **필러도 문서(청크 아님) 단위로 샘플링**: 청크 단위로 무작위 추출하면 대부분의 문서가 문단 한두 개만 걸리고 나머지는 빠지는 식으로 조각나는데, 이러면 나중에 검색된 문단 주변 맥락을 넓혀서 보여주는 parent 확장 전략을 테스트할 수 없다. 그래서 "이 문서는 코퍼스에 있다/없다"가 항상 명확하도록 문서를 통째로 포함하거나 아예 빼는 방식을 택함. (기존 데이터셋이 문서 단위로 청킹돼 있어서가 아니라, 확장 컨텍스트 활용 가능성을 위한 선택)
- 최종: 문서 31,514개, 청크 정확히 **200,000개** (전체 corpus의 약 13.5%)

재현성 검증(멀티시드 평균±표준편차) 및 코퍼스 크기별 성능 포화(saturation) 분석을 위해, 생성 스크립트는 `--seed`/`--target` 인자를 받아 여러 변형을 만들 수 있도록 되어 있음(예: `python build_reduced_corpus.py --seed 43 --target 200000`).

## 6. 생성된 그래프

`EDA/figures/` 폴더에 있는 그래프 5개, 각각의 의미는 다음과 같다.

### `queries_len_hist.png`
x축은 질문 하나의 **글자 수**, y축은 그 글자 수 구간에 속하는 **질문 개수**. 전체 질문 2,761개를 대상으로 함. 분포가 10–30자 구간에 뾰족하게 몰려 있고(평균 19.7자, 표준편차 10.9), 60자를 넘는 질문은 거의 없음 — Ko-MIRACL의 질문은 대부분 한 문장짜리 짧은 검색 쿼리 형태라는 뜻. 80–130자대에 드문드문 튀는 값들은 예외적으로 긴 서술형 질문.

### `qrels_score_dist.png`
x축은 `train`/`dev` 스플릿, y축은 해당 스플릿의 qrels **행 개수**를 score(0=비관련, 1=관련)별로 누적한 막대그래프. train이 dev보다 규모가 훨씬 크고(12,767행 vs 3,057행), 두 스플릿 모두 비관련(score=0) 비율이 관련(score=1)보다 월등히 많음(전체 기준 약 5.3:1) — 즉 qrels 자체가 "관련 문서 1–2개 + 비관련 후보 다수"를 판정해둔 랭킹 평가용 구조라는 걸 보여줌.

### `docs_per_query_hist.png`
x축은 **쿼리 하나당 관련 문서(score=1) 개수**, y축은 그 개수를 가진 **쿼리의 수**. 평균 2.33개, 최대 12개. 대부분의 쿼리가 관련 문서 1–2개만 가지고 있고, 여러 개(5개 이상)를 가진 쿼리는 소수 — 즉 검색 난이도가 쿼리마다 다르며, 정답이 여러 개인 쿼리는 예외적임을 보여줌.

### `corpus_len_hist_sample.png`
x축은 corpus 청크(문단) 하나의 **글자 수**, y축은 **청크 개수**. corpus 전체 148만 개를 다 로드하기엔 무거워서, 스트리밍으로 순차 스캔하며 30개당 1개씩(약 49,558개) 뽑은 샘플 기준. 분포가 짧은 쪽(수십–수백 자)에 집중돼 있음 — Ko-MIRACL corpus는 위키백과 문서를 문단 단위로 잘게 쪼갠 형태(문서당 평균 3.4개 청크)라, 각 청크가 문서 전체가 아니라 짧은 문단 하나 정도 분량이라는 뜻.

### `subset_len_hist.png`
x축은 우리가 추출한 **subset(5,000개) 각 문서의 글자 수**, y축은 **문서 개수**. 평균 271.1자(표준편차 195.2)로 corpus 전체 샘플 분포와 비슷한 모양 — 즉 관련도/hard negative 기준으로 걸렀다고 해서 텍스트 길이 자체가 전체 corpus 대비 유별나게 짧아지거나 길어지지 않았음을 확인용으로 비교하는 그래프.
