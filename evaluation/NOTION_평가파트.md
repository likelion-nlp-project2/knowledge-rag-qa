# 블록 A — 5.5절에 삽입

## 5.5 리트리버 파인튜닝 — 전/후 비교

- **동기**: 사전학습 임베딩은 대상 도메인·언어에 정합되지 않아 정답 문서를 상위로 올리지 못함 → **(query, 정답문서) 쌍으로 대조학습**해 도메인에 정합시킴.
- **방법**: `jhgan/ko-sroberta-multitask` + **MultipleNegativesRankingLoss** (in-batch negative — 배치 안의 나머지 문서를 자동으로 네거티브로 사용). 학습 질문 300개(train split), 1 에폭, 배치 32.
- **평가 조건**: dev 정답 질의 **213개 전수** / 코퍼스 **20만 청크**(E1과 동일 규모) / **리랭커·질의재작성 없이 dense 단계만** 측정.
    - 리랭커를 얹으면 파인튜닝 전 모델의 실수를 리랭커가 교정해 **개선폭이 가려짐** → 리트리버를 재려면 리트리버만 남긴다.
- **신뢰성**: 학습 seed **3회**(42/43/44) 반복 후 **평균 ± 표준편차**. 평가셋 213개는 seed와 무관하게 고정.

### 결과 (seed 3회 평균 ± 표준편차)

| 지표 | 파인튜닝 전 | 파인튜닝 후 | Δ | 개선% |
| --- | --- | --- | --- | --- |
| **Hit@1** | 0.5368 ± 0.0044 | 0.6119 ± 0.0221 | **+0.0751 ± 0.0203** | **+14.0%** |
| **nDCG@10** | 0.5992 ± 0.0023 | 0.6610 ± 0.0183 | **+0.0618 ± 0.0162** | **+10.3%** |
| MRR@10 | 0.6466 ± 0.0010 | 0.7022 ± 0.0160 | +0.0556 ± 0.0160 | +8.6% |
| nDCG@5 | 0.5637 ± 0.0010 | 0.6291 ± 0.0204 | +0.0654 ± 0.0195 | +11.6% |
| Recall@10 | 0.6976 ± 0.0023 | 0.7419 ± 0.0131 | +0.0443 ± 0.0118 | +6.4% |
| Recall@1 | 0.3260 ± 0.0044 | 0.3713 ± 0.0177 | +0.0453 ± 0.0134 | +13.9% |
| Hit@5 | 0.7903 ± 0.0022 | 0.8106 ± 0.0080 | +0.0203 ± 0.0089 | +2.6% |
| Hit@10 | 0.8372 ± 0.0022 | 0.8545 ± 0.0066 | +0.0172 ± 0.0080 | +2.1% |

### 결론

- **12개 지표 × seed 3회 = 36개 비교가 전부 개선**(음수 Δ 0개). 개선폭이 표준편차의 **2.2~3.8배**로, 학습 무작위성으로 설명되지 않는 크기.
- **개선이 상위 순위에 집중** — @1 지표가 **+14.0%** 인 반면 Hit@10은 +2.1%. 즉 파인튜닝은 *"못 찾던 문서를 찾게"* 하기보다 **"이미 찾은 것 중 정답을 위로 끌어올린다"**. 정답을 가깝게·나머지를 멀게 미는 in-batch negative 대조학습의 메커니즘과 정확히 일치.
- **실질 효과**: 213개 질문 중 **약 16개**가 새로 "1등에서 정답"을 찾게 됨 (114.3개 → 130.3개).
- **평가셋 고정 확인**: `before` 표준편차가 최대 **0.0044** — 학습 전 성능이 seed와 무관하게 거의 동일 → 평가셋 213개가 흔들리지 않았음을 수치로 확인.

> ⚠️ **해석 한계 (발표 시 먼저 밝힐 것)**
> 파인튜닝 대상은 `ko-sroberta`이나 **실제 검색 API는 `bge-m3`**(3.2절 채택). 따라서 이 결과는 **"파인튜닝 기법이 리트리버를 개선한다"**는 증명이며, *"우리 시스템 검색이 14% 좋아진다"*로 옮길 수 없다.
> 참고로 파인튜닝 후 ko-sroberta의 Hit@1(0.6119)은 bge-m3 사전학습(0.6432)과의 **격차를 71% 메웠다** — 파라미터가 5배 작은 모델(110M vs 568M)이 22스텝 학습으로 근접. 다만 전 지표에서 여전히 bge-m3가 우위이며, **모델이 다르므로 직접 비교는 삼갈 것.**

> 📌 **후속 과제 (파이프라인)** — ① 파인튜닝된 모델이 저장되지 않아 재사용 불가(`run_compare`에 `model.save()` 없음) ② `torch.manual_seed` 부재로 같은 seed에도 학습이 재현되지 않음 ③ 총 22스텝이라 로깅 주기에 걸리지 않아 **학습 곡선 생성 불가**

# 블록 B — 10절로 삽입 (9절 뒤)

## 10. 평가 (Evaluation)

> 이 절은 평가 **체계**를 정의한다. 각 단계의 개별 비교 실험(임베딩 3.2 · HyDE 4 · 리랭커 6.2 · 파인튜닝 5.5 · 프롬프트 9.2)은 모두 여기서 정의한 평가셋과 지표를 사용한다.

### 10.1 평가 설계 — 왜 검색과 생성을 분리하는가

RAG는 **검색 품질이 생성 품질의 상한**이다. 두 단계를 하나의 점수로 합치면, 점수가 낮을 때 **검색이 못 찾은 것인지 생성이 망친 것인지 진단할 수 없다.** 따라서 검색 지표와 생성 지표를 **끝까지 분리해서** 측정한다.

**평가셋 — dev 정답 질의 213개 전수**

- Ko-miracl **dev split**에서 정답(score>0)이 있는 질의 **전부**. 표본이 아닌 **전수**이므로 샘플링 편향이 정의상 0.
- **train split은 평가에서 제외** — 리트리버 파인튜닝 학습에 쓰이므로 평가에 섞으면 **데이터 누수**(학습에 본 질문으로 성능을 재는 셈).
- 검색·생성·파인튜닝 비교가 **모두 동일한 이 213개**를 사용 → 실험 간 비교 가능.

**분포 점검 2축** (평가셋이 특정 유형에 치우치지 않았는지 확인)

| 축 | 방법 | 결과 |
| --- | --- | --- |
| 질문 길이 | 평가셋 vs 전체 질의의 길이 분포 대조 (`check_evalset_dist.py`) | 유의미한 편향 없음 |
| 카테고리 | 정답 문서의 `doc_type`(DB 기존 값) 기준 분포 (`check_evalset_topics.py`) | 9종 분포, 장소 편중 관찰 |

> 카테고리는 우리가 만든 분류가 아니라 **DB에 이미 있는 `doc_type`** 을 사용해 자의성 논란을 배제. 질문이 아닌 **정답 문서** 기준으로 라벨링 — 질문으로 검색해 라벨을 붙이면 검색 성능이 라벨을 오염시킴(순환).

**신뢰성 — seed 반복 기준**

| 대상 | 반복 | 이유 |
| --- | --- | --- |
| 파인튜닝 전/후 비교 | **seed 3회** 평균±표준편차 | 매 회 새로 학습하므로 실제 변동 발생 |
| 생성 평가 | **1회** | API + temperature=0 이라 결정적 — 반복해도 표준편차가 0에 수렴, 판정 비용만 증가 |
| 검색 베이스라인 | **1회** | 학습 없음 + 코퍼스·모델 고정 → 완전 결정적 |

### 10.2 리트리버 평가

**지표 정의** (`rag/metrics.py`, k = 1 / 5 / 10)

| 지표 | 묻는 것 | 특징 |
| --- | --- | --- |
| **Hit@k** | 상위 k개에 정답이 **하나라도** 있나 | 가장 관대. 서비스 체감에 가까움 |
| **Recall@k** | 정답 문서 **전체 중 몇 %** 를 담았나 | 질문당 정답이 평균 2개 이상이라 @1에서는 낮게 나오는 게 정상 |
| **MRR@k** | 첫 정답이 **몇 등**인가 (역순위) | 상위 노출 능력 |
| **nDCG@k** | 정답들이 **얼마나 위쪽에** 몰렸나 | 순위 가중 종합 지표 |

> k=1에서는 Hit·MRR·nDCG가 수학적으로 같은 값이 된다(1등에서 맞히면 역순위도 nDCG도 1). 지표 구현이 정상임을 확인하는 신호로도 쓰인다.

**E1. 베이스라인 검색 성능**

- 조건: `bge-m3` **사전학습 그대로** / 질의재작성·리랭커·파인튜닝 **없음** / 원 질문으로 검색
- 평가셋 213 전수 · 코퍼스 20만 청크 (`evaluation/run_retrieval_eval.py`)

| 지표 | @1 | @5 | @10 |
| --- | --- | --- | --- |
| Recall | 0.3962 | 0.7001 | **0.8444** |
| MRR | 0.6432 | 0.7507 | **0.7585** |
| nDCG | 0.6432 | 0.6762 | **0.7266** |
| Hit | 0.6432 | 0.8920 | **0.9484** |

**해석**

- **Hit@10 = 94.8%** — 213개 중 **202개**가 상위 10개 안에서 정답 확보. 20만 개 중에서 고른 결과로 양호.
- **Hit@1 = 64.3%** — **137개**는 1등에서 바로 정답.
- **MRR@10 = 0.7585** — 1등에서 못 맞힌 경우에도 평균 3등 부근에서 정답 등장.
- **정답을 못 찾은 11개 질문** → 정성 분석(검색 미스 사례) 대상.
- 이 값이 **이후 모든 개선 실험의 기준선**이 된다.

> 3.2절(임베딩 비교)·6.2절(리랭커 비교)은 후보 풀 2만으로 약식 측정한 **모델 간 상대 비교**이므로, 위 절대 수치와 직접 비교할 수 없다.

**E2. 파인튜닝 전/후 비교** → **5.5절 참조** (12개 지표 전부 개선, Hit@1 +14.0%)

### 10.3 생성 평가

생성은 **두 층**으로 나눠 측정한다. 규칙 기반은 판정 LLM 없이 결정적이라 재현이 보장되고, RAGAS는 규칙으로 못 잡는 의미 품질을 본다. **서로를 교차 검증**하는 구조.

**1층 — 규칙 기반 (결정적, `rag/gen_metrics.py`)**

| 지표 | 정의 |
| --- | --- |
| **인용 정확성** | 답변의 `[문서 n]`이 실제 제시된 문서 범위 안인가 (범위 유효성 + 인용 부착률) |
| **환각율** | 근거가 없는데 기권하지 않고 답을 지어낸 비율 |
| **기권 적절성** | 답할 수 있으면 답하고(과잉 기권 ✗), 근거 없으면 기권했는가 — **양방향** |

**2층 — RAGAS (LLM-as-judge)**

- **faithfulness**(근거 충실도) · **answer relevancy**(답변 관련성), judge **`gpt-5.4-mini` 고정** (생성 모델 `gpt-4o-mini`보다 상위 → 채점이 의미를 가짐, 계열이 달라 자기 채점 편향 없음).
- `context_precision`은 RAGAS가 정답 **'문장'**(ground truth)을 요구하는데 Ko-miracl에는 정답 **'문서'** 만 있어 **사용 불가** → 생성 레이블 구축 후 가능.

**환각 측정 설계 — 무근거 대조군**

환각율과 기권 적절성은 **근거가 없는 상황을 인위적으로 만들어야만** 측정된다. 같은 213개 질문을 두 조건으로 실행한다.

| 조건 | 인덱스 | 올바른 동작 |
| --- | --- | --- |
| **answerable** | 정답 문서 **포함** | 답변해야 함 (기권하면 과잉 기권) |
| **무근거 A** | 그 213개의 정답 문서를 **제외** | 기권해야 함 (답하면 환각) |
| 무근거 B (참고) | 코퍼스 밖 질문 5개 | 기권. 라벨이 추정이라 **정량 집계에서 제외**, 정성 예시로만 |

> **RAGAS faithfulness는 환각율을 대체하지 못한다.** faithfulness는 "**주어진 문서**에 충실한가"만 보므로, 검색이 헛다리를 짚어 엉뚱한 문서가 들어와도 그 문서에 충실하면 점수가 높다. "근거가 없을 때 정직하게 모른다고 하는가"는 **무근거 조건에서만** 측정된다.

**결과**

> [입력 필요: 213 전수 실측 — 1층 3지표 + 2층 RAGAS 2지표]
> **선행 조건**: 채택된 프롬프트가 `fewshot`(9.2절)인데 평가 경로(`rag/generation.py`)는 `basic`과 동일 → **fewshot 연결 후 실측**해야 제품과 같은 조건이 됨.

### 10.4 평가의 한계

- **seed 3회는 통계 검정이 아니다** — "일관된 방향을 확인했다"까지가 정확한 표현. 표본 3개로 유의성을 주장하지 않는다.
- **파인튜닝 모델과 서비스 모델 불일치** — E2는 `ko-sroberta`, 서비스는 `bge-m3` (5.5절 한계 참조).
- **단일 데이터셋·단일 도메인** — Ko-miracl 한국어 위키. 다른 도메인 일반화는 미검증.
- **judge와 생성기가 같은 계열(OpenAI)** — 자기 채점 편향은 없으나 완전 독립은 아님 → 1층 규칙 기반 지표와 교차 확인으로 보완.
- **평가셋 규모 불일치** — 절별로 다름(3.2·6.2: 200개 / 9.2: 100건 / E1·E2: 213 전수). 각 실험 내부의 상대 비교는 유효하나, **절대 수치를 절 간에 비교하지 말 것.** 특히 9.2는 train split을 포함하므로 절대값 인용 시 주의.
