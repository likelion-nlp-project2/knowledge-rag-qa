## E1. 베이스라인 검색 성능 (2026-07-23)

계획서 4.1 **E1** / 5절 7/23 산출물 "1차 성능(Recall/MRR/nDCG/Hit)".

### 실험 설정

| 항목 | 값 |
|---|---|
| 임베딩 모델 | `BAAI/bge-m3` (사전학습 그대로, 파인튜닝 없음) |
| 검색 방식 | dense 검색만 (코사인 유사도) |
| 리랭커 | **사용 안 함** |
| 질의 재작성 | **사용 안 함** (원 질문 그대로 검색) |
| 평가셋 | Ko-miracl dev 정답(score>0) 쿼리 **전수 213개** |
| 코퍼스 | 축소 코퍼스 **200,000 청크** |
| 실행 | `evaluation/run_retrieval_eval.py` (Colab GPU, 임베딩 18분 53초) |

### 결과

| 지표 | @1 | @5 | @10 |
|---|---|---|---|
| **Recall** | 0.3962 | 0.7001 | 0.8444 |
| **MRR** | 0.6432 | 0.7507 | 0.7585 |
| **nDCG** | 0.6432 | 0.6762 | 0.7266 |
| **Hit** | 0.6432 | 0.8920 | 0.9484 |

### 해석

- **Hit@10 = 94.8%** — 213개 질문 중 **202개**가 상위 10개 안에서 정답 문서를 찾았다. 20만 개 중에서 고른 결과로는 양호하다.
- **Hit@1 = 64.3%** — **137개** 질문은 1등에서 바로 정답을 찾았다.
- **MRR@10 = 0.7585** — 1등에서 못 찾은 경우에도 평균적으로 3등 부근에서 정답이 나온다.
- **Recall@1 = 0.3962** — 낮아 보이지만 정상이다. 질문당 정답 문서가 평균 2개 이상인데 문서를 1개만 뽑았으므로 절반 이하만 담긴다. Recall@10에서 0.8444로 오르는 것이 정확한 동작.
- **@1에서 MRR·nDCG·Hit이 모두 동일(0.6432)** — 1등에서 맞히면 역순위도 nDCG도 1이 되므로 수학적으로 강제된 값. 지표 구현이 정상이라는 확인 신호.
- **정답을 못 찾은 질문 11개** — 계획서 E5(정성 분석, 검색 미스 사례) 대상.

### 이 숫자의 위치

앞으로의 모든 개선 실험(파인튜닝, 질의 재작성, 프롬프트 전략)이 **이 값 대비 얼마나 나아졌는지**로 이야기된다. 발표의 기준선.

> **주의:** E1(bge-m3)은 E2의 "파인튜닝 전(before)"과 **다른 모델**이다. E2는 `jhgan/ko-sroberta-multitask`를 학습시키므로 before 값도 E1과 다르게 나온다. 두 표를 같은 줄에 섞어 읽지 말 것.

## E2. 리트리버 파인튜닝 전/후 비교 (진행 중)

계획서 4.1 **E2** / 5절 7/24~25 산출물 "compare 결과".

### 실험 설정

| 항목 | 값 |
|---|---|
| 베이스 모델 | `jhgan/ko-sroberta-multitask` (E1의 bge-m3와 다름) |
| 학습 방식 | in-batch negative 대조학습 (MultipleNegativesRankingLoss) |
| 학습 질문 | 300개 (train split) |
| 에폭 | **1** (계획서 기본 설정 준수) |
| 반복 | 학습 seed **3회** (42 / 43 / 44) → 평균 ± 표준편차 |
| 평가셋 | E1과 동일한 dev 전수 **213개** (seed 무관하게 고정) |
| 코퍼스 | **200,000** (E1과 동일 규모로 정렬) |
| 실행 | `evaluation/run_finetune_compare_eval.py` |

### 결과 (2026-07-23, seed 3회 평균 ± 표준편차)

| 지표 | 파인튜닝 전 | 파인튜닝 후 | Δ | 개선% |
|---|---|---|---|---|
| Recall@1 | 0.3260 ± 0.0044 | 0.3713 ± 0.0177 | **+0.0453 ± 0.0134** | +13.9% |
| MRR@1 | 0.5368 ± 0.0044 | 0.6119 ± 0.0221 | **+0.0751 ± 0.0203** | +14.0% |
| nDCG@1 | 0.5368 ± 0.0044 | 0.6119 ± 0.0221 | **+0.0751 ± 0.0203** | +14.0% |
| Hit@1 | 0.5368 ± 0.0044 | 0.6119 ± 0.0221 | **+0.0751 ± 0.0203** | +14.0% |
| Recall@5 | 0.5986 ± 0.0016 | 0.6453 ± 0.0160 | **+0.0467 ± 0.0164** | +7.8% |
| MRR@5 | 0.6403 ± 0.0009 | 0.6962 ± 0.0161 | **+0.0559 ± 0.0161** | +8.7% |
| nDCG@5 | 0.5637 ± 0.0010 | 0.6291 ± 0.0204 | **+0.0654 ± 0.0195** | +11.6% |
| Hit@5 | 0.7903 ± 0.0022 | 0.8106 ± 0.0080 | **+0.0203 ± 0.0089** | +2.6% |
| Recall@10 | 0.6976 ± 0.0023 | 0.7419 ± 0.0131 | **+0.0443 ± 0.0118** | +6.4% |
| MRR@10 | 0.6466 ± 0.0010 | 0.7022 ± 0.0160 | **+0.0556 ± 0.0160** | +8.6% |
| nDCG@10 | 0.5992 ± 0.0023 | 0.6610 ± 0.0183 | **+0.0618 ± 0.0162** | +10.3% |
| Hit@10 | 0.8372 ± 0.0022 | 0.8545 ± 0.0066 | **+0.0172 ± 0.0080** | +2.1% |

### 결론 — H2 지지

**12개 지표 전부 개선(음수 Δ 0개)** 되었고, seed 3회 모두 같은 방향이었다.

- **개선폭이 표준편차의 2.2~3.8배** — 흔들림보다 개선이 훨씬 크므로 우연으로 보기 어렵다.
- **상위 순위에서 개선이 가장 크다** — @1 지표가 +14%인 반면 Hit@10은 +2.1%. 파인튜닝이
  "못 찾던 문서를 찾게 만든다"기보다 **"이미 찾은 것 중 정답을 위로 끌어올린다"**는 뜻이다.
  in-batch negative 대조학습의 전형적인 효과.
- **`before std`가 최대 0.0044로 매우 작다** — 학습 전 성능이 seed와 무관하게 거의 같다는 뜻이며,
  평가셋 213개가 seed에 흔들리지 않고 고정됐음을 확인해 준다(계획서 4.3 "동일 평가셋 고정").

### 한계 — 발표 시 먼저 밝힐 것

**E2는 `ko-sroberta`를 파인튜닝했지만, 실제 검색 API와 E1 베이스라인은 `bge-m3`를 쓴다.**
따라서 이 결과는 "파인튜닝 기법 자체가 리트리버를 개선한다"는 것을 보여주지만,
"우리 시스템의 검색이 이만큼 좋아진다"로 바로 옮길 수는 없다.

- bge-m3(568M)는 ko-sroberta(110M)보다 5배 크고 이미 강해서, 같은 학습량으로 개선하기 더 어렵다
- 팀 논의 필요: (a) bge-m3 파인튜닝 시도, (b) 현 결과를 기법 검증으로 한정해 발표, (c) 시스템 임베딩 통일

참고로 파인튜닝 후 ko-sroberta의 Hit@1(0.6119)은 E1의 bge-m3(0.6432)에 근접했으나,
Hit@10은 0.8545 대 0.9484로 여전히 차이가 있다. **모델이 다르므로 직접 비교는 삼갈 것.**

### 부가 관찰 (팀 공유 사항)

실행 중 확인된, 평가 외 영역의 개선 필요 사항:

1. **파인튜닝된 모델이 저장되지 않는다** — `run_compare`에 `model.save()`가 없어 학습 결과가
   seed마다 버려진다. 데모나 후속 평가에 재사용하려면 필요. (`rag/cli.py`, 파이프라인 담당)
2. **학습이 재현되지 않는다** — `torch.manual_seed(seed)`가 없어 DataLoader shuffle이 제어되지
   않는다. 같은 seed로도 결과가 달라진다. (`rag/cli.py`, 파이프라인 담당)
3. **학습 곡선을 만들 수 없다** — 총 22스텝인데 로깅 주기에 걸리지 않아 loss가 기록되지 않는다.
   계획서 7/24~25 산출물 "학습 곡선"이 현 구조로는 불가. (`rag/finetune.py`, 파이프라인 담당)

### 참고 — 학습량 사전 확인

본 실행 전 소규모로 학습량과 지표 변화의 관계를 확인했다(숫자 자체는 규모가 작아 보고용이 아님).

| 학습 질문 | 에폭 | 총 스텝 | 결과 |
|---|---|---|---|
| 30 | 1 | 3 | 12개 지표 Δ 전부 0.0 |
| 200 | 3 | 45 | Recall@1 +8.6%, Hit@1 +4.8% 등 전 지표 개선 |
| **300 (본 실행)** | **1** | **22** | — |

본 실행은 22스텝으로, 개선이 확인된 45스텝의 절반 수준이다. Δ가 작게 나올 경우 "파인튜닝이 효과 없다"가 아니라 **학습량 부족**을 먼저 의심해야 한다. 에폭 수는 파이프라인 담당(영주님) 설정이므로, 필요 시 위 근거를 공유하고 조정을 협의한다.

## 생성 평가 (2026-07-24)

계획서 4.3 **생성 주 지표**. RAG가 검색한 문서로 **생성한 답변**의 품질을 잰다.
(E1·E2는 검색 성능이고, 이건 그 뒤 LLM 생성 단계다 — 담당·도구가 다르다.)

### 실험 설정

| 항목 | 값 |
|---|---|
| 생성 모델 | `gpt-4o-mini` (temperature=0 → 결정적, seed 1회로 충분) |
| 프롬프트 | **fewshot** (프롬프트 A/B 실험 채택 — faithfulness/answer_relevancy 둘 다 우위) |
| 판정 모델 (RAGAS judge) | `gpt-5.4-mini` |
| 평가셋 | dev 정답 쿼리 **213개** × 2조건(answerable / 무근거A) + 무근거B 5 |
| 코퍼스 | **200,000** 청크 (E1·E2와 동일 규모) |
| 실행 | `evaluation/run_generation_eval.py` (Colab A100) |

지표를 두 층으로 나눈다:
- **1층 = 규칙 기반**(정규식 + answerable 라벨). LLM 판정 없이 결정적으로 계산 → seed에 무관하게 재현.
- **2층 = RAGAS**(LLM 판정). 의미 수준 품질.

### 1층 — 규칙 기반 지표 (LLM 판정 없음, 결정적)

| 지표 | 값 | 의미 |
|---|---|---|
| **citation_validity** | **1.0000** | 답변의 모든 `[문서 n]` 인용이 유효 범위(1..n) 안 |
| **citation_coverage** | **1.0000** | 비기권 답변 전부가 인용을 1개 이상 담음 |
| **over_abstention_rate** | **0.0516** | answerable 질문 중 잘못 기권한 비율(213개 중 11개) — 과잉 거절 거의 없음 |
| **무근거B 기권율** | **1.0000** | 코퍼스 밖 질문(날씨·주가 등)엔 100% 기권 |

- **인용 규율이 완벽하다** — 답한 384건 전부가 존재하는 문서 번호만 인용했고(validity), 비기권 답변은 예외 없이 근거를 표시했다(coverage). fewshot 프롬프트가 인용 형식을 강제한 효과.
- **과잉 기권이 5%뿐** — 답할 수 있는 질문은 거의 다 답한다. 지나치게 몸을 사리지 않는다.
- **코퍼스에 진짜 없는 질문(무근거B)은 100% 기권** — 근거 없이 지어내지 않는다.

### 2층 — RAGAS 의미 지표 (판정기 `gpt-5.4-mini`, 0~1)

| 지표 | 값 | 대상 |
|---|---|---|
| **faithfulness** (충실도) | **0.9491** | answerable 비기권 202건 |
| **answer_relevancy** | **0.8515** | 〃 |
| **context_recall** | **0.8912** | 정답 라벨 있는 193건 |
| **answer_correctness** | **0.6870** | 〃 |

**정답 라벨(ground_truth) 구축** — Ko-miracl엔 정답 문서만 있고 정답 답변 문장이 없어, context_recall·answer_correctness(4컬럼 지표)를 계산하려면 정답 답변이 필요하다. `gpt-5.4`가 **정답 문서만을 근거로** 답변을 생성해 라벨 200건을 만들었다(정답 문서에 답이 없는 13건 제외). 검색 결과가 아니라 정답 문서로 만들어, 검색 실패가 정답에 흡수되지 않게 했다. **라벨·판정·생성 모델을 `gpt-5.4` / `gpt-5.4-mini` / `gpt-4o-mini`로 분리**해 자기 채점 편향을 막았다. 라벨 자체의 근거성은 자동 검사(gold 문서 기준 faithfulness)로 **평균 0.917**이었다.

### 해석

- **충실도 0.9491 — 답변이 검색 문서에 충실하다.** 근거 밖 내용을 지어내는 경우가 적다는 뜻으로, 1층의 완벽한 인용 규율과 같은 방향이다.
- **context_recall 0.8912 는 E1 `Hit@10=0.9484`와 정합한다.** 검색이 정답 근거 문서를 대부분 가져온다는 것을, 이번엔 **독립적으로 만든 정답 라벨로 재확인**한 셈이다.
- **answer_correctness 0.6870 은 나머지보다 낮다 — "31% 틀렸다"가 아니다.** RAGAS answer_correctness는 사실 F1(0.75) + 의미 유사도(0.25)인데, 우리 답변은 `[문서 n]` 인용과 완결 문장("영조는 조선의 제21대 왕**입니다** [문서 1]")인 반면 정답 라벨은 단답("조선의 제21대 왕**이다**")이다. 이 **길이·문체 차이가 유사도를 깎고, 추가 문장이 사실 F1에서 false positive로 잡힌다.** 표현 차이가 상당 부분 섞인 값이다.

### 한계 — 발표 시 밝힐 것

1. **4컬럼 지표는 silver label 기반이다.** 정답 라벨을 LLM(`gpt-5.4`)이 만들고 다른 LLM(`gpt-5.4-mini`)이 채점하는 구조라, 사람 검수를 거친 gold label과는 신뢰 수준이 다르다. 모델 3층 분리와 근거성 자동 검사(0.917)로 편향은 낮췄으나, 사람 검수는 수행하지 않았다.
2. **Faithfulness는 생성 품질과 청킹 품질이 섞인 지표다.** 정답 청크에 질문의 핵심어가 없는 경우(청킹이 주어를 잘라낸 경우) 라벨 근거성 미달률이 14.1%로, 있는 경우 7.8%의 1.8배였다. 같은 영향이 시스템 답변 채점에도 작용하므로, 충실도 점수엔 청킹 손실이 일부 포함돼 있다.

```

  1번 셀   코드 받기
```
