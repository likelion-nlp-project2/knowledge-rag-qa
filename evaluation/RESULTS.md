## E1. 베이스라인 검색 성능 (2026-07-23)

계획서 4.1 **E1** / 5절 7/23 산출물 "1차 성능(Recall/MRR/nDCG/Hit)".

### 실험 설정

| 항목 | 값 |
|---|---|
| 임베딩 모델 | `BAAI/bge-m3` (사전학습 그대로, 파인튜닝 없음) |
| 검색 방식 | dense 검색만 (코사인 유사도) |
| 리랭커 | **사용 안 함** |
| 질의 재작성 | **사용 안 함** (원 질문 그대로 검색) |
| 평가셋 | Ko-miracl dev 정답(score>0) 쿼리 **전수 213개** |
| 코퍼스 | 축소 코퍼스 **200,000 청크** |
| 실행 | `evaluation/run_retrieval_eval.py` (Colab GPU, 임베딩 18분 53초) |

### 결과

| 지표 | @1 | @5 | @10 |
|---|---|---|---|
| **Recall** | 0.3962 | 0.7001 | 0.8444 |
| **MRR** | 0.6432 | 0.7507 | 0.7585 |
| **nDCG** | 0.6432 | 0.6762 | 0.7266 |
| **Hit** | 0.6432 | 0.8920 | 0.9484 |

### 해석

- **Hit@10 = 94.8%** — 213개 질문 중 **202개**가 상위 10개 안에서 정답 문서를 찾았다. 20만 개 중에서 고른 결과로는 양호하다.
- **Hit@1 = 64.3%** — **137개** 질문은 1등에서 바로 정답을 찾았다.
- **MRR@10 = 0.7585** — 1등에서 못 찾은 경우에도 평균적으로 3등 부근에서 정답이 나온다.
- **Recall@1 = 0.3962** — 낮아 보이지만 정상이다. 질문당 정답 문서가 평균 2개 이상인데 문서를 1개만 뽑았으므로 절반 이하만 담긴다. Recall@10에서 0.8444로 오르는 것이 정확한 동작.
- **@1에서 MRR·nDCG·Hit이 모두 동일(0.6432)** — 1등에서 맞히면 역순위도 nDCG도 1이 되므로 수학적으로 강제된 값. 지표 구현이 정상이라는 확인 신호.
- **정답을 못 찾은 질문 11개** — 계획서 E5(정성 분석, 검색 미스 사례) 대상.

### 이 숫자의 위치

앞으로의 모든 개선 실험(파인튜닝, 질의 재작성, 프롬프트 전략)이 **이 값 대비 얼마나 나아졌는지**로 이야기된다. 발표의 기준선.

> **주의:** E1(bge-m3)은 E2의 "파인튜닝 전(before)"과 **다른 모델**이다. E2는 `jhgan/ko-sroberta-multitask`를 학습시키므로 before 값도 E1과 다르게 나온다. 두 표를 같은 줄에 섞어 읽지 말 것.

## E2. 리트리버 파인튜닝 전/후 비교 (진행 중)

계획서 4.1 **E2** / 5절 7/24~25 산출물 "compare 결과".

### 실험 설정

| 항목 | 값 |
|---|---|
| 베이스 모델 | `jhgan/ko-sroberta-multitask` (E1의 bge-m3와 다름) |
| 학습 방식 | in-batch negative 대조학습 (MultipleNegativesRankingLoss) |
| 학습 질문 | 300개 (train split) |
| 에폭 | **1** (계획서 기본 설정 준수) |
| 반복 | 학습 seed **3회** (42 / 43 / 44) → 평균 ± 표준편차 |
| 평가셋 | E1과 동일한 dev 전수 **213개** (seed 무관하게 고정) |
| 코퍼스 | **200,000** (E1과 동일 규모로 정렬) |
| 실행 | `evaluation/run_finetune_compare_eval.py` |

### 결과

*(실행 완료 후 표 삽입 — before / after / Δ / 개선%, 각각 평균과 표준편차)*

### 읽는 법

- **Δ가 양수인가** → 파인튜닝이 검색을 개선했는가 (가설 H2)
- **std가 Δ에 비해 작은가** → 세 번 모두 비슷하게 개선됐는가. Δ +0.05에 std 0.01이면 신뢰할 만하고, std가 Δ와 비슷하면 단정하기 어렵다

### 참고 — 학습량 사전 확인

본 실행 전 소규모로 학습량과 지표 변화의 관계를 확인했다(숫자 자체는 규모가 작아 보고용이 아님).

| 학습 질문 | 에폭 | 총 스텝 | 결과 |
|---|---|---|---|
| 30 | 1 | 3 | 12개 지표 Δ 전부 0.0 |
| 200 | 3 | 45 | Recall@1 +8.6%, Hit@1 +4.8% 등 전 지표 개선 |
| **300 (본 실행)** | **1** | **22** | — |

본 실행은 22스텝으로, 개선이 확인된 45스텝의 절반 수준이다. Δ가 작게 나올 경우 "파인튜닝이 효과 없다"가 아니라 **학습량 부족**을 먼저 의심해야 한다. 에폭 수는 파이프라인 담당(영주님) 설정이므로, 필요 시 위 근거를 공유하고 조정을 협의한다.
